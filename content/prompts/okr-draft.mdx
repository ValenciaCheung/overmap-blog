prompt: |
  你是一位资深产品 / 工程领导,深谙 Google OKR / Andy Grove 的 OKR 方法论。

  请基于我的"想做的事"产出一个完整 OKR。

  ## OKR 规范(严格遵守)

  ### Objective(O)
  - 一句话,有"野心"(stretch goal)
  - 不可量化(eg. "成为... / 让... 体验更好"),量化在 KR 里
  - 包含"为什么这件事现在重要"的隐含价值

  ### Key Results(3 个,刚好 3 个)
  - 每个都是"数字 + 时间窗口" 形式
  - 70-80% 完成的程度算"good"(不要写一定能完成的 KR)
  - 必须可观测(eg. "提升用户活跃" 不行,"周活从 X 涨到 Y" 才行)
  - 3 个 KR 之间不能高度相关(避免一个挂全挂)

  ### 验收标准
  对每个 KR,明确:
  - **怎么测量**(指标名 / 数据来源)
  - **基线在哪**(当前数值)
  - **谁负责**(责任人)

  ### 月度里程碑(分 4 个月或 3 个月)
  - Month 1: ...
  - Month 2: ...
  - Month 3: ...
  (如果 Q 是 3 个月,3 个里程碑;6 个月就 6 个)

  ### 风险 / 假设
  - 列 2-3 个"如果 X 不成立,这个 OKR 就失败"的关键假设
  - 给 mitigation 建议

  ## 约束
  - Objective 不能用空词 ("revolutionize" / "world-class")
  - KR 数字要具体合理(基于现实基线 1.5-2x 拉伸)
  - 月度里程碑要"前重后轻"(前 2 个月要拿到 50%+ 进度)

  ## 我的想做的事
  {{一段话描述目标场景。例如:"我们这个季度要让 AI 工具站从 0 涨到有真实流量"}}

  - 团队规模 / 资源:{{eg. 我 + 1 个外包设计师 / 周 20 小时}}
  - 时间窗口:{{eg. 2026 Q3,3 个月}}
  - 当前基线(数据):{{eg. 当前周访问 0}}
---

## 用法

季度初 / 启动新项目时用。**OKR 是给团队对齐用的,不是给老板看的**。

输出后:
1. 跟团队 / 利益相关方对一遍 KR 是否合理
2. 把 3 个 KR 放进每周 sync 议程
3. 月底回顾一次完成度,调整后两个月里程碑

## 改写思路

- **个人 OKR** → 调整约束 "1 个 O + 2 个 KR 就够,不要 3 个,容易分心"
- **公司级** → 改成 "Objective 必须能传达给所有员工 / 投资人,30 字以内"
- **NSM (North Star Metric) 模式** → 简化成 "一个核心指标 + 输入 / 输出 / 反向 leading indicators"

## 坑点

- AI 默认会写"安全 KR"(很容易完成的),约束里强制"拉伸 KR"
- 团队没有数据基线时,先做 1 周数据收集再写 KR,不要瞎拍数字
- OKR 不是 KPI —— OKR 是 quarterly,KPI 是持续的。约束模型按 quarterly 视角
